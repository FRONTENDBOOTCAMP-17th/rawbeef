## 2. 잘한 점

### Tailwind CSS 활용
- Tailwind 유틸리티 클래스를 활용해서 별도의 CSS 파일 없이 스타일링을 잘 하고 있어요.
- `hover:`, `transition` 같은 인터랙션 스타일도 잘 적용했습니다.

### LocalStorage 활용
- `admin_category.html`과 `admin_song.html`에서 데이터를 LocalStorage에 저장/불러오기 하는 것이 잘 구현되어 있어요.
- 페이지를 새로고침해도 데이터가 유지되는 것은 사용자 경험에 좋습니다.

### CRUD 기능 구현
- 카테고리 관리: 추가, 수정, 삭제가 모두 동작합니다.
- 노래 관리: 추가, 수정, 삭제 + 점수 기반 정렬까지 구현했어요.
- `confirm()`으로 삭제 전 확인을 받는 것도 좋은 습관입니다.

### 드래그 앤 드롭
- `admin_category.html`에서 카테고리 순서를 드래그로 변경할 수 있게 만든 것은 도전적이고 훌륭해요!
- `mouseenter`/`mouseleave`로 드래그 핸들(= 버튼)에서만 드래그가 시작되도록 한 것도 세심합니다.

### 기본기
- `lang="ko-KR"` 설정, `charset="UTF-8"`, `viewport` 메타 태그 등 HTML 기본 설정이 잘 되어 있어요.
- Pretendard 폰트를 `preload`로 미리 로드하는 것도 좋은 성능 최적화 방법입니다.

## 3. 개선하면 좋을 점

### 3-1. Tailwind CSS CDN vs Vite 플러그인 (중요도: 높음)

**현재 상태:** `soyu/` 폴더의 HTML 파일들은 CDN 방식으로 Tailwind를 사용하고 있어요.
```html
<!-- soyu/ 파일들에서 사용 중인 방식 -->
<script src="https://cdn.tailwindcss.com"></script>
```

**문제점:**
- 프로젝트에 이미 Vite + Tailwind 플러그인(`@tailwindcss/vite`)이 설정되어 있는데, `soyu/` 폴더의 파일들은 이걸 사용하지 않고 있어요.
- CDN 방식은 개발할 때 편하지만, 실제 배포할 때는 파일 크기가 크고 느려질 수 있어요.
- 두 가지 방식이 섞여 있으면 나중에 혼란스러워질 수 있어요.

**추천:** `soyu/` 폴더의 페이지들도 Vite 빌드 시스템에 포함시키는 것이 좋아요. `vite.config.js`의 `findAllHtmlFiles` 함수에 `soyu` 폴더도 추가해보세요.

```js
// vite.config.js 에서
input: {
  index: path.resolve(__dirname, "index.html"),
  ...findAllHtmlFiles(path.resolve(__dirname, "src")),
  ...findAllHtmlFiles(path.resolve(__dirname, "soyu")),  // 이 줄 추가
},
```

### 3-2. 사이드바 코드 반복 (중요도: 높음)

**현재 상태:** 4개의 HTML 파일 모두에 같은 사이드바 코드가 복사-붙여넣기 되어 있어요.

```html
<!-- 이 코드가 4개 파일에 똑같이 있음 -->
<aside class="w-1/5 bg-white border-r-4 border-black flex flex-col p-6 gap-6">
  <a href="./admin_open.html" ...>...</a>
  <a href="./admin_category.html" ...>...</a>
  <a href="./admin_song.html" ...>...</a>
  <a href="./admin_request.html" ...>...</a>
</aside>
```

**왜 문제일까?**
- 메뉴 항목을 하나 추가하거나 이름을 바꾸려면 4개 파일을 모두 수정해야 해요.
- 파일이 많아질수록 실수할 확률이 높아져요.

**추천:** 사이드바를 별도의 JavaScript 함수나 파일로 분리해서 재사용하면 좋아요. 아직 배우는 단계이니 지금 당장은 괜찮지만, 이런 반복되는 부분을 "컴포넌트"로 분리하는 것이 프로젝트가 커질수록 중요해집니다.

### 3-3. `<a>` 태그 안에 `<button>` 넣기 (중요도: 높음)

**현재 상태:**
```html
<a href="./admin_open.html" class="block w-full">
  <button class="w-full py-4 ...">관리자메뉴</button>
</a>
```

**문제점:** HTML 규칙상 `<a>` 태그 안에 `<button>`을 넣는 것은 허용되지 않아요. 둘 다 "클릭할 수 있는 요소"이기 때문에 충돌이 생겨요.

**추천:** 둘 중 하나만 사용하세요.

```html
<!-- 방법 1: <a> 태그를 버튼처럼 보이게 스타일링 -->
<a href="./admin_open.html"
   class="block w-full py-4 border-2 border-black font-bold text-center hover:bg-gray-200">
  관리자메뉴
</a>

<!-- 방법 2: <button>을 쓰고 JavaScript로 이동 -->
<button onclick="location.href='./admin_open.html'"
        class="w-full py-4 border-2 border-black font-bold hover:bg-gray-200">
  관리자메뉴
</button>
```

### 3-4. 모달의 `hidden`과 `flex` 충돌 (중요도: 중간)

**현재 상태:**
```html
<div id="modal" class="fixed inset-0 bg-black hidden flex items-center justify-center z-50">
```

**문제점:** `hidden`은 `display: none`을 적용하고, `flex`는 `display: flex`를 적용해요. 두 클래스가 동시에 있으면 어느 쪽이 이길지 예측하기 어려워요.

Tailwind CSS에서는 뒤에 쓴 클래스가 항상 이기는 게 아니라, CSS 내부의 우선순위에 따라 결정돼요.
현재는 `hidden`이 우선 적용되어서 잘 동작하는 것처럼 보이지만, Tailwind 버전이 바뀌면 깨질 수 있어요.

**추천:** `hidden`을 제거할 때 `flex`를 함께 추가하는 방식으로 변경하세요.

```html
<!-- HTML에서 flex 제거 -->
<div id="modal" class="fixed inset-0 bg-black hidden items-center justify-center z-50">
```

```js
// 모달 열기 - hidden 제거 + flex 추가
modal.classList.remove('hidden');
modal.classList.add('flex');

// 모달 닫기 - flex 제거 + hidden 추가
modal.classList.add('hidden');
modal.classList.remove('flex');
```

### 3-6. `admin_category.html` - 저장 누락 (중요도: 중간)

**현재 상태:** 카테고리를 추가/수정할 때 `saveCategory()`를 호출하지 않아서, 새로고침하면 추가한 내용이 사라져요.

```js
// saveCategoryBtn 이벤트 리스너 안에서
if (title) {
  if (editIndex !== null) {
    category[editIndex].title = title;
    editIndex = null;
  } else {
    category.push({ title: title });
  }
  renderCategory();
  // <-- 여기에 saveCategory(); 가 빠져 있어요!
  modal.classList.add('hidden');
```

**추천:** `renderCategory()` 뒤에 `saveCategory();`를 추가하세요.

### 3-7. `admin_song.html` - score가 0일 때 저장이 안 됨 (중요도: 중간)

**현재 상태:**
```js
if (title && artist && score) {
```

**문제점:** JavaScript에서 숫자 `0`은 `false`로 취급돼요. 그래서 score에 0을 입력하면 "내용이 입력되지 않았습니다!" 알림이 뜨게 됩니다.

**추천:**
```js
// score가 0이어도 통과하도록 수정
if (title && artist && score !== null && !isNaN(score)) {
```

### 3-8. `admin_request.html` - `<title>` 오류 (중요도: 낮음)

**현재 상태:**
```html
<title>관리자 카테고리</title>
```

이 페이지는 "노래 신청 관리" 페이지인데, `<title>`이 "관리자 카테고리"로 되어 있어요. 아마 다른 파일을 복사하면서 수정이 안 된 것 같습니다.

**추천:**
```html
<title>노래 신청 관리</title>
```

### 3-9. `admin_open.html`, `admin_request.html` - 메인 콘텐츠 영역 없음 (중요도: 낮음)

**현재 상태:** 이 두 페이지는 사이드바만 있고, 메인 콘텐츠 영역(`<main>`)이 없어요. 아직 개발 중인 것 같지만, 빈 페이지가 나오면 사용자가 "페이지가 고장났나?" 하고 혼란스러울 수 있어요.

**추천:** 간단하게라도 "준비 중입니다" 같은 안내 문구를 넣어두면 좋아요.

```html
<main class="flex-1 bg-white p-8 flex items-center justify-center">
  <p class="text-gray-400 text-xl">준비 중입니다</p>
</main>
```

### 3-10. `<label>`과 `<input>` 연결하기 (중요도: 중간)

**현재 상태:**
```html
<label class="block font-black mb-1">제목</label>
<input type="text" id="titleInput" ... />
```

**문제점:** `<label>`과 `<input>`이 `for`/`id`로 연결되어 있지 않아요. 연결하면 label을 클릭했을 때 자동으로 input에 포커스가 가고, 스크린 리더(시각장애인용 도구) 사용자도 어떤 입력 칸인지 알 수 있어요.

**추천:**
```html
<label for="titleInput" class="block font-black mb-1">제목</label>
<input type="text" id="titleInput" ... />
```
